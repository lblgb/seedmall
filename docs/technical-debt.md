# SeedMall 技术债与后续补强点

这份文档记录当前为了学习节奏先简化处理、后续需要补强的设计点。

## 秒杀 Reservation Key 生命周期

当前 `seedmall-seckill` 使用 Redis key 防止同一用户重复抢同一商品：

```text
seckill:reservation:{productId}:{userId}
```

现状：

- 首次秒杀请求通过 `SETNX` 写入 reservation key。
- key TTL 当前为 30 分钟。
- 重复请求命中 key 后直接返回“已在排队中”，不重复扣库存、不重复发 MQ。
- 库存不足时会主动删除 reservation key，允许用户后续重试。

当前简化点：

- 抢购成功后 reservation key 只依赖 30 分钟 TTL 自动过期。
- 订单创建失败、订单取消、活动结束等状态还没有反向释放或延长 reservation key。
- 如果业务要求“每人每活动只允许买一次”，TTL 应该跟活动结束时间对齐，不能固定 30 分钟。

后续补强方向：

- 订单消费失败时释放 reservation key。
- 订单超时取消时按业务规则决定是否释放 reservation key。
- 秒杀活动结束后统一清理活动相关 reservation key。
- 将固定 TTL 改为活动结束时间或活动配置中的防重复窗口。

## 订单创建与商品库存扣减补偿

当前订单服务在新订单落库后，通过 Feign 调用商品服务扣减数据库库存：
```text
OrderService -> ProductStockClient -> ProductFeignClient -> ProductService -> product.stock
```

现状：
- 订单唯一索引保证同一用户同一商品不会重复创建订单。
- 只有新订单成功落库后才会调用商品服务扣减数据库库存。
- 命中重复订单时直接返回已有订单，不会重复扣减商品数据库库存。
- 商品服务使用 `stock >= quantity` 条件更新，避免数据库库存被扣成负数。

当前简化点：
- 如果订单已经落库，但远程扣减商品库存失败，当前还没有自动取消订单或补偿库存链路。
- 如果商品扣减成功，但后续订单状态流转失败，当前也没有统一的库存回滚策略。
- Redis 预扣库存、数据库真实库存、订单状态之间还没有完整的最终一致性闭环。

后续补强方向：
- 引入本地消息表或 RocketMQ 事务消息，保证“订单创建事件”和“库存扣减事件”可恢复。
- 增加订单状态机，将扣库存失败的订单标记为失败或待补偿。
- 增加定时补偿任务，扫描异常订单并重试扣库存或关闭订单。
- 将 Redis 秒杀库存和数据库库存做定期对账，发现差异后输出修复任务。
